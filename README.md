# Cplusplus-Program                                                                                                                                                             
# Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?                                                                                                             For this project, the client was The Gaming Room, a company that wanted a web-based game application that could support multiple users and devices. Their main goal was to create a scalable, cross-platform game similar to “Draw It or Lose It,” where teams can compete in real time. The software needed to handle user authentication, game sessions, and data storage efficiently while making sure everything runs smoothly across different operating systems. Overall, the focus was on designing a system that was reliable, easy to expand, and user-friendly.                                                                                                                              
# What did you do particularly well in developing this documentation?                                                                                                       One thing I think I did well in developing this documentation was clearly breaking down the system requirements and matching them to design choices. I made sure to explain why certain technologies and architectures were chosen, instead of just listing them. This helped connect the client’s needs to the actual system design. I also stayed organized with sections like system architecture, constraints, and recommendations, which made the document easier to follow.                                                                                                                    
# What about the process of working through a design document did you find helpful when developing the code?                                                                                                                                                      Working through the design document was actually really helpful when it came to understanding how the code would be structured. It forced me to think ahead about things like scalability, platform compatibility, and performance before even starting development. Instead of just jumping into coding, I had a clear plan, which made everything feel more intentional and less confusing.                                                                                                                
# If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?                                                                                                                                      If I could go back and revise one part of my work, I would improve the system architecture diagrams and explanations. I would make them more detailed and possibly include visuals to better show how different components interact. That would make it easier for both technical and non-technical stakeholders to understand how the system works.                                                                                                                                                            
# How did you interpret the user’s needs and implement them into your software design?                                                                                                                                                                 Why is it so important to consider the user’s needs when designing?When interpreting the user’s needs, I focused on what the client actually wanted the experience to be like—simple, accessible, and functional across multiple platforms. I translated those needs into design decisions like using a web-based architecture and ensuring the system could handle multiple users at once. Considering the user’s needs is important because, at the end of the day, the software is built for them. If their needs aren’t met, the system won’t be successful no matter how well it’s coded.                                                                                          
# How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?                                                                                                    My overall approach to designing the software was to start with the requirements, then move into system design, and finally evaluate different technologies that could meet those needs. I used strategies like breaking the system into smaller components, comparing platform options, and thinking about future scalability. In the future, I would continue using these techniques, but I would also spend more time prototyping and testing early designs to catch potential issues sooner.
